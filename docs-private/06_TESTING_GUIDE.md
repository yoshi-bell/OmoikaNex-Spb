# SNSアプリ 戦略的テスト指針 (Strategic Testing Guide)

本プロジェクトでは、AIを「テストを書かせる助手」ではなく、**「仕様の穴を検討する検証者（QAエグゼクティブ）」**として扱い、これまでに培った**「技術的な実装基準」**に基づいた、精度の高い品質保証を目指す。

> [!IMPORTANT]
> **テストの物理的な実行手順（バックアップ、リセット、リストア、実行コマンド等）については、必ず `docs-private/03_WORKFLOW.md` を参照すること。** ここでは「テストコードの品質」と「検証戦略」のみを定義する。

## ⚔️ セクション1：三大テスト戦略 (AI-Driven Strategies)

開発効率と多角的な品質検証を両立させるため、以下の3つの戦略を遵守するよう努める。

### 戦略1: エッジケース探索プロンプト (実装前の検討)
正常系（Happy Path）の実装に入る前に、必ずAIに「不具合が起きやすいエッジケース」を網羅的にリストアップさせる。
- **観点:** 境界値（0, 負数, 極大値）、サロゲートペア（絵文字）、不正な入力型（null, undefined）、複雑な組み合わせ。
- **反映:** 抽出されたリストを `TEST_CASES.md` に反映し、実装時の指針とすること。

### 戦略2: プロパティベーステストの考え方の導入 (性質の検証)
個別の値だけでなく、「どのような入力であっても常に満たすべき性質（不変条件）」を定義し、検証を試みる。
- **例:** 「タイムラインをどれだけスクロールしても、同じツイートが重複して表示されないこと」「フォローを解除したら、相手のフォロワーリストから自分が消えていること」。
- **テンプレート:** AIに「プロパティベーステストの考え方で、この機能の不変条件を検討して」と依頼する。

### 戦略3: AI対AIの「敵対的テスト生成」 (検証の強化)
自ら書いたコードの「弱点」を探るため、客観的な視点でテストを生成し、検証を深める。
- **プロンプト:** 「あなたは厳格なQAエンジニアです。このコードの論理矛盾やセキュリティ上の懸念を突く、失敗するテストケースを検討してください」。
- **修正:** 検討されたケースをパスするようにコードを補強することで、システムの堅牢性を高めることを目指す。

## 🛠️ セクション2：技術的実装基準 (Technical Implementation Standards)

テストの「保守性」を担保するため、これまでの開発経験から得られた以下の実装詳細を遵守するよう努める。

### 1. Backend: Database / RLS テスト (Supabase / pgTAP 等)
- **環境:** Supabase Local CLI 環境を使用し、テスト間の独立性を保つことを目指す。
- **検証:** データベースレベルでの `CHECK` 制約の動作確認と、Edge Functions が意図したデータを返すかを検証する。
- **認可ガード (RLSの検証):** Row Level Security (RLS) が正しく機能しているか。「他人のリソース（ツイート等）を削除・更新できないか」「未ログインユーザーが保護されたデータを読めないか」という異常系（権限エラー）の確認を重視する。
- **優先順位:** RLSによる認可ガードと、DB制約によるバリデーション異常系の確認を、テストの最優先事項と位置づけることを目指す。

### 2. Frontend: 状態・コンポーネントテスト (Vitest / Testing Library)
- **DOMクエリ:** `within` や `getByRole` を優先使用し、HTML構造やCSSクラスの変更に左右されにくい、安定したテストの記述を目指す。
- **モック戦略:** `msw` (Mock Service Worker) を活用し、APIのレスポンス（Loading / Error / Success）による UI の状態遷移を網羅的に検証することを目標とする。
- **カスタムフック:** 複雑なステート管理は UI から切り離し、`renderHook` でロジック単体を精査する。

### 3. E2E テスト (Playwright)
- **ワークフロー:** ログインから投稿、いいね、ページ遷移までの一連のユーザー操作が完走することを検証する。
- **楽観的UI更新:** 「いいね」等の操作において、API完了を待たずにUIが更新される挙動と、API失敗時のロールバックが視覚的に正しく行われるかの確証を目指す。

### 4. テスト要件書との同期とトレーサビリティ (Requirement Traceability)
- **ID の完全一致:** テストコード内の `describe` や `it` の冒頭には、必ず `TEST_CASES.md` で定義された ID（例: `ID 4-11`）を明記すること。
- **文言の同期:** テストの内容（説明文）は、要件書の「テスト内容」および「期待挙動」と可能な限り文言を一致させ、コードと仕様の乖離をゼロにすること。
- **目的:** これにより、AI と人間が共創する環境において「どの仕様がどのテストで守られているか」を瞬時に追跡可能（トレース可能）にし、将来のリファクタリング時の安全性を極大化する。

## 💎 セクション3：型安全性と実利的なモック

- **any の抑制:** テストコードにおいても原則として `any` を避け、`unknown` や具体的な型定義による安全性の確保に努める。
- **実利的な例外規定:** 
    - 外部ライブラリの型定義が極めて困難な場合に限り、実利的な判断として `any` の使用を検討する。
    - その際は必ず `// eslint-disable-next-line @typescript-eslint/no-explicit-any` を添え、使用理由をコメントで明記すること。
- **アサーションの意図:** 失敗時に「何が仕様と異なっていたのか」が伝わるような、意図を重視したテストコード（Intent-focused Assertions）の記述を心がける。
