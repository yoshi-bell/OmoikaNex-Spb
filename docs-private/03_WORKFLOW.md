# SNSアプリ 開発ワークフロー (Workflow & Checklists)

## 開発ルール (Rules for Execution)

### 動作確認ルール (Verification Standards)

ロードマップの各タスクを「完了」とする前に、**必ず** 以下の検証を行い、正常動作を確証すること。

- **バックエンド (Supabase / DB):**
    - **データ整合性:** DBの中身を直接確認し、データが正しく保存・更新されているか。
    - **型定義とRLS:** `database.types.ts` の型と実際のデータ構造が一致しているか。RLS (Row Level Security) により不正なアクセスが弾かれているか。
    - **API機能:** Supabase Client 経由でのリクエストに対して、正しいデータまたはエラー（PostgRESTエラー等）を返しているか。
- **フロントエンド (Next.js/UI):**
    - **型安全:** ZodバリデーションおよびReact Queryからのデータ型が正しく認識されているか。
    - **状態管理:** Zustandストアのグローバルな状態が予測通りに更新されているか。
    - **楽観的UI:** いいね処理などのAPIレスポンスを待たないUI更新が正常に機能しているか。
- **環境整合性の検証 (Environment Parity):**
    - **Local Production Build:** リリース前には必ず `npm run build && npm run start` を実行し、本番モード (`NODE_ENV=production`) で動作確認を行うこと。特に画像最適化やビルド時の型エラーをこの段階で完全に排除する。
- **ネットワーク通信:**
    - Networkタブでブラウザへ送信されるデータの最適化（過剰なリレーション先の読み込み防止など）がされているか。
    - 本番モードにおいて、Next.js による画像最適化がエラーなく機能し、適切なサイズで配信されているか。

### 戦略的テスト実行フロー (Data Protection & Test Flow)

E2Eテスト（Playwright）等で本物のDBを使用する際、開発データを保護しながらクリーンな検証を行うため、以下の「三段階フロー」を遵守すること。

#### 1. 開発データのバックアップ (Backup Phase)
- **public スキーマ:** `npx supabase db dump --data-only -f supabase/dev_backup_public.sql` により一般データを抽出。
- **auth.users スキーマ:** Supabase の仕様上 `dump` に含まれにくいため、現在の `auth.users` の挿入SQLを手動またはスクリプトで `supabase/dev_backup_auth.sql` として保存する。
- **重要:** テスト開始前に必ずこの 2 ファイルが正常に生成されたことを確認すること。

#### 2. テストの実行 (Execution Phase)
- **指針の再確認 (Crucial):** 実装を開始する前に、**必ず `docs-private/06_TESTING_GUIDE.md` を読み込み**、そこに定義された「三大テスト戦略」および「技術的実装基準」を最新の正解として遵守すること。
- **環境構築:** `seed.sql` をテスト専用のデータ（`test_seed.sql` 相当）に一時的に差し替え、`npx supabase db reset` でクリーンなテスト環境を構築。

- **検証:** `TEST_CASES.md` に基づく Vitest / MSW / Playwright テストを完走させる。
- **実装:** テストコードは各機能の近傍（`__tests__/`）に配置し、Repository層のエラーマッピングを含めて検証する。


#### 3. 開発環境の復元 (Restore Phase)
- **復旧:** テスト完了後、バックアップした `dev_backup_auth_safe.sql` および `dev_backup_public_safe.sql` を順次適用（または `seed.sql` を元に戻してリセット）し、開発データを完全に復元する。
- **プロ仕様のリストア:** トリガーによる PK 重複エラー（on_auth_user_created 等）を防ぐため、リストア時は必ず `SET session_replication_role = 'replica';` を実行してトリガーを一時的に無効化すること。
- **整合性確認:** 復元後にブラウザでログインし、以前のデータが正しく表示されることを目視確認する。

### DB 変更の標準サイクル (DB Migration Workflow)

スキーマ（テーブル、カラム、RLS）を変更する際は、必ず以下の手順を遵守すること。

1. **SQL 作成:** `supabase/migrations/` 内に新規マイグレーションファイル（`.sql`）を作成する。
2. **反映:** `npx supabase db push` コマンドでクラウド上のデータベースに反映させる。
3. **型同期:** `npm run codegen` を実行し、変更後の構造に基づいた `database.types.ts` を再生成する。
4. **実装:** 更新された型定義を用いてフロントエンドの実装・修正を行う。

### 📝 記録プロセス実行ルール (Mandatory Protocol)

開発状況を記録し、状態を永続化する際は、**必ず** 以下の 4 ステップを順守した後に `git push` を実行してください。

#### **Step 1: 開発ログの記録 (History)**
- **対象ファイル:** `docs-private/07_DEV_LOG.md`
- **アクション:** 実装内容、技術的判断（Why）、テスト結果、および次回の展望を追記する。
- **注意:** `write_file` ではなく、既存のログを保護するため `replace` による末尾追記を徹底すること。

#### **Step 2: 復旧マニュアルの同期 (Recovery)**
- **対象ファイル:** `docs-private/01_AGENT_RECOVERY_MANUAL.md`
- **アクション:** `07_DEV_LOG.md` の更新に基づき、末尾の「想定される結論」セクション（フェーズ、完了事項、次タスク）を最新の状態へ書き換える。

#### **Step 3: ワークフローの更新 (Status)**
- **対象ファイル:** `docs-private/03_WORKFLOW.md` (本書)
- **アクション:** 
    1. 「Phase 7: 品質保証」等のロードマップのチェックボックス (`[ ]` -> `[x]`) を更新。
    2. 「【現在進行中のタスク】」セクションの内容を、次の作業者が即座に再開できるレベルまで具体化する。

#### **Step 4: Git コミット & プッシュ (Persistence)**
- **言語:** 日本語で記述 (`feat:`, `fix:`, `docs:`, `refactor:` 等のプレフィックスを使用)。
- **署名 (Findy対策):** コミットメッセージの末尾に、必ず空行を挟んで以下の署名を含めること。
  ```text
  Co-authored-by: Gemini CLI <gemini-cli@google.com>
  ```
- **実行:** 上記 3 つのドキュメントとコードの変更を一括でコミットし、プッシュする。
  ```bash
  git add .
  git commit -m "feat: [機能名] の実装とテスト完了、およびドキュメントの同期"
  git push origin [branch-name]
  ```
