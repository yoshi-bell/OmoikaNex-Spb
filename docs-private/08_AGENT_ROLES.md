# 🤖 AI駆動開発: アーキテクトと実装エージェントの役割分担

本プロジェクト（OmoikaNex-Spb）では、複雑化するコンテキストウィンドウを保護し、最高品質のコードを生み出すために、AIの役割を意図的に2層に分離（Tiered AI Architecture）して開発を進めます。

## 1. 役割の定義 (Roles Definition)

### 🗡️ Tier 1: アーキテクト・エージェント (Senior Architect)

- **ミッション:** 「何を（What）、どういうルールで（Why）作るべきか」の定義と、成果物の「検閲（Review）」。
- **禁止事項:** 手を動かして環境構築（例: `npm install`）や、大量のボイラープレートコード（設定ファイル等）を直接記述すること。
- **責務:**
    1. ユーザーの要件を聞き出し、プロジェクトの憲法（`02_RULES...`, `04_CODING...`）と照らし合わせる。
    2. 実装エージェントが**コピペで実行できるレベルの「詳細設計書（プロンプト）」**（例: 更新された `TEST_CASES.md` など）を作成する。
        - **[重要ルール]** 指示書を作成する際は、必ず「どのテストコードを、どのファイルパスに配置すべきか（例: `src/features/.../__tests__/xxx.test.tsx`）」の対応表を明記し、実装エージェントからアーキテクチャ上の迷いを完全に排除すること。
    3. 実装エージェントが記述したコード（GitのDiff）をレビューし、アーキテクチャの劣化や「テストの抜け漏れ（エッジケース・プロパティ）」がないか厳格に検査する。

### 🛡️ Tier 2: 実装・エージェント (Implementation Agent / Coder)

- **ミッション:** アーキテクトから渡された「設計書・指示書」を、迅速かつ正確に「動くコード（How）」に翻訳すること。
- **禁止事項:** 指示書にないアーキテクチャの変更（勝手なライブラリ選定、ディレクトリ構造の改変など）を独断で行うこと。
- **責務:**
    1. 指示書（例えば `TEST_CASES.md` に書かれた MSW のモック条件やエッジケース）を愚直にテストコード（Vitest / Playwright）として実装する。
    2. コンパイルエラーやリンターエラー（Oxlint/TypeScript等）が発生した場合、自己修復ループを回してエラーを潰し切る。
    3. テストが Green (パス) になるまで泥臭く作業を完遂させる。

## 2. 協働ワークフロー (Collaboration Workflow)

機能追加やテストフェーズ（Phase 7等）において、ユーザーと2つのAIは以下のサイクルで協働します。

1. **[User -> Architect] 要件定義:** ユーザーがアーキテクトにタスクを相談する。
2. **[Architect] 設計＆プロンプト生成:** アーキテクトが関連ファイル（`TEST_CASES.md` 等）を更新し、「実装エージェントへの指示書（What & Why）」を完成させる。
3. **[User] プロンプトの受け渡し:** ユーザーがアーキテクトの作った指示書（またはファイルパス）を、実装エージェントが入っている別のウィンドウ（あるいはCursor等のインラインAI）に渡す。
4. **[Implementation] コーディング:** 実装エージェントが指示通りに手を動かし（How）、環境構築やテストコードを書き上げる。
5. **[Architect] コードレビュー:** 実装完了後、アーキテクトが `git diff` や `git log` を読み込み、要件（エッジケース等のカバー）が満たされているか最終承認を行う。

## 3. この分離パラダイムの目的 (Why we do this)

1. **コンテキストウィンドウの純度維持:** アーキテクトの脳内（コンテキスト）から、エラー解決や `npm` のログといった「ノイズ」を排除し、常にシステム全体を俯瞰させるため。
2. **敵対的テスト（Adversarial Testing）の実現:** アーキテクト（QA・攻撃者）が意地悪なテストケースを考え、実装エージェント（防御者）がそれを満たすコードを書くことで、AI同士の検証プロセスを強制的に構築するため。
